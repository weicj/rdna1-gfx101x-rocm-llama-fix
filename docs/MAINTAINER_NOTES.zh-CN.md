# 维护者说明：项目定位与后续分支策略

这份文档不是给公开读者看的首页内容，而是给项目维护者自己做取舍时用的内部说明。

## 1. 这个项目为什么适合先作为独立仓库发布

当前最有价值的部分，其实不是某几行源码补丁，而是下面这些已经沉淀出来的“工程结论”：

- `W5500 / Navi14 / gfx1012` 在老平台上的实际 bring-up 路径
- `MEC 123 -> 156` 这个跨过 `KFD` 门槛的关键点
- `Linux 6.8 + ROCm 6.3.3` 为什么是第一条稳定推理通路
- `ROCm 7.2.1` 为什么需要额外补 `gfx1012` 的 `rocBLAS/Tensile` 资产
- `llama.cpp` 在 `W5500` 上真正跑通了哪些现代模型底座、性能到什么程度、限制在哪里
- `TTFT` 为什么会慢，以及哪些慢是“后端慢”，哪些其实是“reasoning / streaming 语义问题”

这些内容单独就已经足够构成一个原创项目。

换句话说，这个项目当前最强的地方是：

- 一份别人很难一次性写全的真实工程说明
- 一套别人不容易在旧平台上亲自踩出来的结论

所以它完全适合作为一个独立仓库先发出去。

## 2. 为什么现在不急着把它定义成 llama.cpp fork

如果现在就直接开 `llama.cpp` fork，会有两个问题：

1. 主题会被稀释
   - 读者会以为项目重点是维护一棵源码树
   - 但你这套工作的真正亮点其实是 `W5500` 在 `ROCm 6/7` 上的实战打通

2. 维护成本会被提前放大
   - 一旦 fork 了，就会天然背上“同步 upstream、维护 patch、处理 merge”的长期成本
   - 但现在真正需要长期维护的源码差异，其实还没大到那个程度

所以更合理的策略是：

- **先发独立项目仓库**
- **后面真的形成长期 patch 链，再开 fork**

## 3. 哪些情况下值得单独开一个 llama.cpp fork

只有在下面这些内容真正沉淀为长期资产时，才值得把 `llama.cpp` fork 独立出来：

- `Q1_0 / Q1_0_g128` 这类实验性量化路径准备长期维护
- `gfx101x/gfx1012 + ROCm7` 的兼容修补需要持续保留源码差异
- 你准备长期维护一套 `W5500` / `gfx101x` 友好的构建分支
- 需要把一整套部署补丁、量化核、CI 逻辑固定在源码树里

如果后面真到这一步，fork 的方向也应该尽量收敛，不要变成一个泛用大杂烩。

当前已定下的仓库名方向是：

- `rdna1-gfx101x-rocm-llama-fix`

它的优点是：

- 明确点出 `RDNA1`
- 明确点出 `gfx101x`
- 明确这是 `ROCm + llama.cpp` 方向
- 同时保留 `fix` 这个“工程适配/修补项目”的味道

不建议再把名字收窄到只剩 `W5500`，因为这套方法论本身已经更接近 `RDNA1 / Navi14 / gfx101x` 通用解法；但 README 正文里仍应明确说明真实验证主样本是 `W5500`。

## 4. 当前更适合的 GitHub 项目定位

当前最强的定位不是“又一个 ROCm 教程”，而是：

> 一个围绕 `RDNA1 / Navi14 / gfx101x` 展开的实战项目：目标是让这类卡在 `ROCm 6 / ROCm 7` 上真正跑起现代大模型，并完整记录固件门槛、内核选择、ROCm 用户态补层、`llama.cpp` 验证、模型矩阵、性能结果与限制条件；当前真实验证主样本是 `Radeon Pro W5500 / gfx1012`。

这个定位的好处是：

- 主题明确
- 原创性强
- 一眼就能看出这是实机工程总结，不是资料拼贴

## 5. 维护建议

现在这套项目更适合按两层维护：

### 对外公开层

- `README`
- `SUPPORTED_BASELINES`
- `EXPERIMENTAL_QUANTIZATION`
- `KNOWN_LIMITATIONS`
- `AGENT_RUNBOOK`

这些内容应该只写给公开读者，不掺“我要不要 fork”“我这个 GitHub 账号要怎么定位”这种维护者自言自语。

### 对内维护层

- 本文件这种 maintainer note
- 未来的发布策略、分支策略、命名策略
- 哪些内容准备沉淀成 fork，哪些继续保留在独立项目里

这两层分开之后，公开仓库会显得专业很多，也更像一个成熟项目。
